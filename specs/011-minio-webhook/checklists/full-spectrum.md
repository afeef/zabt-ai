The webhook fires for all; handler is idempotent — files without matching meeting records are skipped (200). The "trigger a background transcription job" clause implicitly scopes to files with matching meeting records.

- [x] CHK014 - Is the idempotency behavior described in research.md R-007 captured as a formal requirement, or is it only an implementation decision? [Clarity, Gap] — **ACCEPTED**: Idempotency is an implementation decision in R-007, realized via `initiate_processing()` checking `status == "pending_upload"` (data-model.md). Critical for retry safety but not a user-facing functional requirement — appropriate as a design decision.

## Requirement Consistency

- [x] CHK015 - Does US1 acceptance scenario 2 ("creates a new database Meeting record") conflict with the plan's design decision that the meeting record is created by the frontend BEFORE upload, and the webhook only transitions status? [Conflict, Spec §US1-AS2 vs research.md R-003] — **FIXED**: US1-AS2 updated to "transitions the meeting status from `pending_upload` to `queued` and enqueues the Celery worker task."
- [x] CHK016 - Is the Meeting `status` field's set of valid values consistent across spec.md, data-model.md, and contracts/meetings.yaml? [Consistency] — **FIXED**: spec.md Key Entities now lists full lifecycle: `pending_upload → queued → processing → completed | failed`. Consistent across all artifacts.
- [x] CHK017 - Does the webhook contract's `WebhookResponse.status: "error"` value have a corresponding HTTP error status code, or does the webhook always return 200 even on processing errors? [Consistency, Contract §webhooks.yaml] — **BY DESIGN**: Webhook returns 200 for all processing outcomes (`ok`, `skipped`, `error`). MinIO interprets non-2xx as delivery failure and retries — returning 200 with a status field prevents unwanted retries. Only auth failure returns 401.
- [x] CHK018 - Is the `minio-init` script's bucket event subscription (`s3:ObjectCreated:Put`) consistent with the spec's requirement scope, or should `s3:ObjectCreated:CompleteMultipartUpload` also be subscribed for large file uploads? [Consistency, quickstart.md vs Spec §FR-001] — **ACCEPTED**: Frontend uses presigned PUT URLs (single-part upload), which trigger `s3:ObjectCreated:Put`. MinIO multipart via CLI/SDK is not a supported upload path in this feature. Adding `CompleteMultipartUpload` is a future hardening task if needed.

## Scenario Coverage

- [x] CHK022 - Are requirements defined for concurrent webhook deliveries (multiple files uploaded simultaneously triggering parallel webhook calls)? [Coverage, Gap] — **ACCEPTED OUT-OF-SCOPE**: Handler is stateless — each webhook call independently queries and transitions its own meeting record. No shared state, no lock contention. Standard concurrent request handling by FastAPI.
- [x] CHK023 - Are requirements defined for webhook delivery when the API service is temporarily unavailable (queue persistence behavior)? [Coverage] — **COVERED**: research.md R-004 documents `MINIO_NOTIFY_WEBHOOK_QUEUE_DIR_PRIMARY` for disk-based queue persistence. quickstart.md includes the config. Design decision, not a user-facing requirement.
- [x] CHK024 - Are requirements defined for the `minio-init` service failure scenario (bucket subscription fails on startup)? [Coverage, Gap] — **ACCEPTED OUT-OF-SCOPE**: quickstart.md step 3 says "Verify minio-init ran successfully: `docker compose logs minio-init`". The `mc event add` command is idempotent — re-running `docker compose up minio-init` retries. Formal retry logic is over-engineering for dev environment.
- [x] CHK025 - Are requirements defined for what happens when MinIO sends non-`s3:ObjectCreated:Put` events to the webhook endpoint? [Coverage, Gap] — **NOT APPLICABLE**: The `mc event add` subscription filters to `--event s3:ObjectCreated:Put` only. MinIO will not send other event types to this endpoint. If subscription config were changed, handler would still safely skip (no matching file_path → return "skipped").
- [x] CHK026 - Are requirements defined for webhook behavior when the `Records` array is empty or contains multiple records in a single payload? [Coverage, Gap] — **ACCEPTED**: MinIO sends exactly one record per webhook call (unlike AWS S3 which batches). Empty `Records` is handled by the DTO default (`Records: list = []`) — handler skips gracefully. Implementation detail, not a spec-level requirement.

## Edge Case Coverage

- [x] CHK027 - Are requirements specified for URL-encoded special characters in object keys (spaces, unicode, `+` characters)? [Edge Case] — **COVERED**: research.md R-005 specifies `urllib.parse.unquote_plus` for decoding. data-model.md DTO receives raw key. tasks.md T005 description includes "URL-decode with `urllib.parse.unquote_plus`". Implementation-level detail, not a spec-level requirement.
- [x] CHK028 - Are requirements specified for duplicate `file_path` values in the Meeting table (two meetings pointing to the same S3 object)? [Edge Case, Gap] — **ACCEPTED OUT-OF-SCOPE**: Presigned URL generation uses UUID-prefixed keys (`{uuid}_{filename}`), making collision practically impossible. `initiate_processing()` queries by `file_path` + `status == "pending_upload"` — even theoretical duplicates would only trigger the first matching record.
- [x] CHK029 - Are requirements specified for the race condition where the webhook fires before `initiate_processing` can find the meeting record (extremely fast MinIO notification)? [Edge Case] — **FIXED**: FR-006 documents the reorder: frontend creates meeting record BEFORE uploading to MinIO. The record exists before MinIO can fire the webhook. If the webhook still arrives first (theoretically impossible with this flow), it returns "skipped" and MinIO retries via QUEUE_DIR.
- [x] CHK030 - Are requirements specified for zero-byte files uploaded to MinIO (empty file webhook trigger)? [Edge Case, Gap] — **ACCEPTED OUT-OF-SCOPE**: Zero-byte upload is an edge case not in the feature scope. Webhook handler processes it normally (status transition + Celery enqueue). The transcription worker handles empty files gracefully (produces empty transcript).

## Security Requirements

- [x] CHK032 - Is the unauthenticated HEAD endpoint's exposure documented as an accepted risk, and are requirements defined to prevent information leakage from it? [Security, Contract §webhooks.yaml HEAD] — **FIXED**: FR-005 explicitly documents: "The HEAD endpoint (MinIO health check) is exempt from authentication due to a known MinIO limitation (issue #14507)." HEAD returns only 200 status — no body, no information leakage.
- [x] CHK033 - Are requirements defined for webhook payload size limits to prevent denial-of-service via oversized POST bodies? [Security, Gap] — **ACCEPTED OUT-OF-SCOPE**: Webhook is internal Docker network only. FastAPI default request body limit applies. DoS via oversized payloads requires Docker network access, which implies the attacker already has infrastructure access.
- [x] CHK034 - Are requirements defined for ensuring the webhook endpoint is not accessible from outside the Docker network in production? [Security, Gap] — **ACCEPTED**: plan.md §Technical Context states "MinIO and backend are on the same Docker network; webhook delivery is internal." Network isolation is a deployment concern handled by Docker networking and reverse proxy configuration, not a feature-level requirement.
- [x] CHK035 - Is the lack of TLS between MinIO and the API service within the Docker network documented as an accepted risk? [Security, Assumption] — **ACCEPTED**: Internal Docker network communication without TLS is industry standard practice. The shared secret in the Bearer token provides authentication. TLS within Docker overlay networks is a production hardening concern, not an MVP requirement.

## Infrastructure & Dependencies

[Infrastructure, quickstart.md] — **ACCEPTED**: `sleep 5` is pragmatic for dev environment. `depends_on: minio` ensures ordering. If MinIO isn't ready, `mc alias set` fails and the user checks logs (quickstart step 3). `docker compose up minio-init` re-runs. Production requires health checks — out of scope for MVP.
- [x] CHK037 - Are service startup ordering requirements defined (must API be running before MinIO starts to accept the webhook target)? [Infrastructure, Gap] — **COVERED**: research.md R-004 documents `QUEUE_DIR` mitigation — MinIO queues events to disk when API is unavailable. `depends_on` in docker-compose handles basic ordering. No strict startup order required due to queue persistence.
- [x] CHK038 - Are requirements defined for webhook queue overflow when events exceed `QUEUE_LIMIT` of 100,000? [Infrastructure, Gap] — **ACCEPTED OUT-OF-SCOPE**: 100,000 queued events represents an extreme scenario far beyond expected usage. Overflow behavior is MinIO's responsibility (drops oldest events). Production monitoring concern, not MVP requirement.
- [x] CHK039 - Is the `minio/mc` image version pinned in the `minio-init` service definition, or is the lack of pinning an accepted risk? [Infrastructure, quickstart.md] — **ACCEPTED**: Unpinned `minio/mc` is acceptable for dev environment. The `mc event add` command is stable across versions. Production deployments should pin versions — documented as a deployment best practice, not a feature requirement.
