# Research: Export Summary as PDF

**Feature**: `001-export-summary-pdf` | **Date**: 2026-03-04

---

## Decision 1: PDF Generation Library

**Decision**: Use `pdfmake` for client-side PDF generation.

**Rationale**:
- Produces **text-searchable PDFs** (vector text, not rasterised images) — satisfies SC-002 (markdown formatting rendered correctly) and SC-005 (opens in any PDF viewer)
- Built-in `.download(filename)` method triggers automatic browser download — satisfies FR-005
- Full document structure API: header/footer, styles, inline bold/italics, bullet lists — satisfies FR-003
- No SSR complications — works with `'use client'` directive; no dynamic import needed
- Stable, widely-used library (~4M downloads/week) with active maintenance
- Small footprint (~500KB gzipped) vs. alternatives

**Alternatives considered**:

| Option | Verdict | Reason rejected |
|--------|---------|-----------------|
| `@react-pdf/renderer` | Rejected | Requires mapping every markdown element to a React component; larger bundle (~2MB); more complex setup for a simple export |
| `jsPDF` + `html2canvas` | Rejected | Captures DOM as image — PDF is not text-searchable; large file sizes; renders Tailwind CSS inconsistently at canvas resolution |
| `window.print()` + print CSS | Rejected | Requires user to manually select "Save as PDF" in print dialog; cannot control filename programmatically; inconsistent cross-browser |

## Decision 2: Markdown Parsing Strategy

**Decision**: Implement a lightweight line-by-line markdown parser in `pdf-export.ts` that maps to pdfmake content nodes.

**Rationale**:
- The summary markdown is generated by the LLM and follows a predictable structure: ATX headings (`##`), bold (`**text**`), italics (`*text*`), and unordered lists (`- item`)
- A purpose-built parser is ~60 lines of code vs. pulling in a full markdown parser (marked, remark) just for PDF export
- Keeps the dependency footprint minimal
- Covers all elements present in the existing built-in templates (General Summary, Meeting Minutes, Action Items, Retrospective)

**Elements handled**:
- `# / ## / ###` headings → pdfmake styled Text nodes
- `**bold**` / `*italic*` → inline pdfmake bold/italics via regex
- `- item` unordered lists → pdfmake `ul` arrays
- Plain paragraphs → justified Text nodes
- Empty lines → spacer nodes

---

## Decision 3: Entry Point (Where to Place the Export Button)

**Decision**: Add "Download as PDF" as a new item inside the existing three-dot summary options dropdown in `meetings/[id]/page.tsx`, below the existing "Download as .txt" item.

**Rationale**:
- The menu already contains "Copy summary" and "Download as .txt" — PDF export is a natural third option in the same grouping
- No new UI pattern or component is needed — reuses the existing menu item layout
- Keeps the summary header area uncluttered

---

## Decision 4: Filename Sanitization

**Decision**: Strip characters invalid in filenames (`/ : * ? " < > | \`) and replace with hyphens, then append `-summary.pdf`.

**Rationale**: Consistent with the existing `.txt` download filename pattern (`${meeting.title}-summary.txt`).

**Implementation**: Simple regex replace in `pdf-export.ts`:
```ts
const safeTitle = title.replace(/[/\\:*?"<>|]/g, "-").trim();
const filename = `${safeTitle}-summary.pdf`;
```
