# Research: Transcription Worker Provider Switch (RunPod / Local)

## Decision 1: RunPod Client Integration Pattern

**Decision**: Use the `runpod` Python SDK with async `/run` + polling (not `/runsync`).

**Rationale**: Transcription of 30+ minute audio files will exceed the `/runsync` default 90-second timeout. The `/run` endpoint returns immediately with a job ID, and we poll via `request.status()` / `request.output(timeout=...)`. This matches our existing async Celery pipeline — the `stage_transcribe` task simply blocks on polling instead of local Whisper execution.

**Alternatives considered**:
- `/runsync` with extended timeout — max configurable is still impractical for long audio; no progress visibility
- Webhook callback — would require a new API endpoint on the backend, complicating the architecture for minimal benefit since we're already inside a Celery task that can block

## Decision 2: Audio File Transfer to RunPod

**Decision**: Pass a presigned download URL from the storage provider (S3/MinIO) to the RunPod handler. The handler downloads the file inside its container.

**Rationale**: Audio files can be hundreds of MB. RunPod `/run` has a 10MB payload limit, ruling out base64 embedding. Presigned URLs are already generated by our storage layer (`storage.get_presigned_download_url()`), and RunPod workers have internet access to download from S3/MinIO.

**Alternatives considered**:
- Base64 in payload — exceeds 10MB limit for any real meeting recording
- Upload to RunPod's storage — no native RunPod storage service; adds complexity
- Shared volume — not possible between VPS and RunPod

## Decision 3: RunPod Handler Architecture

**Decision**: Standalone handler using WhisperX (faster-whisper backend) + pyannote, matching the local WhisperProvider pipeline exactly. Models loaded globally at container start (warm between jobs).

**Rationale**: Using the same WhisperX pipeline ensures output parity with local transcription. Global model loading means subsequent requests skip the ~30s model load (cold start only for first request). RunPod Serverless supports this pattern natively.

**Alternatives considered**:
- Vanilla OpenAI Whisper — slower (no CTranslate2 batching), constitution recommends WhisperX
- Pre-built community workers (e.g., `runpod-workers/worker-faster_whisper`) — don't include pyannote diarization; output format doesn't match our `TranscriptionResult`

## Decision 4: RunPod Response Format

**Decision**: The RunPod handler returns a JSON payload matching our `TranscriptionResult` dataclass structure:

```json
{
  "text": "[SPEAKER_00] Hello...\n[SPEAKER_01] Hi...",
  "language": "en",
  "segments": [
    {
      "start": 0.0,
      "end": 2.5,
      "text": "Hello everyone",
      "speaker": "SPEAKER_00",
      "words": [
        {"word": "Hello", "start": 0.0, "end": 0.5, "speaker_label": "SPEAKER_00", "confidence": null},
        {"word": "everyone", "start": 0.6, "end": 1.2, "speaker_label": "SPEAKER_00", "confidence": null}
      ]
    }
  ],
  "audio_duration_seconds": 1800.0,
  "provider_name": "runpod_whisper",
  "recognition_method": "runpod_whisperx",
  "estimated_cost": 0.0
}
```

**Rationale**: Returning the exact same structure means `RunPodProvider._parse_response()` is a straightforward deserialization with no format transformation needed.

## Decision 5: Polling Configuration

**Decision**: Default polling interval of 5 seconds, timeout of 30 minutes. Both configurable via `RUNPOD_POLL_INTERVAL` and `RUNPOD_TIMEOUT` env vars.

**Rationale**: 5-second interval balances responsiveness against API rate limits (RunPod allows 2000 status checks per 10-second window). 30-minute timeout covers audio files up to ~4 hours on a cold worker.

## Decision 6: Cost Estimate

**Decision**: RunPod GPU cost is approximately $0.00076/second on an A40 GPU ($2.74/hr). A 30-minute meeting takes ~3-5 minutes of GPU time, costing ~$0.14-$0.23 per transcription.

**Rationale**: This is significantly more expensive than local GPU ($0 marginal) but acceptable for a VPS without a GPU. The local provider remains available for cost-sensitive deployments with GPU hardware.

## Decision 7: `runpod` Package Placement

**Decision**: Add `runpod` to the core dependencies in `pyproject.toml` (not the `ml` group). It's a lightweight HTTP client (~2MB) used by the worker to call the RunPod API.

**Rationale**: The `ml` dependency group is for heavy ML packages (PyTorch, WhisperX, pyannote) that require the `worker-base` image. The `runpod` SDK is just an API client and should be available in both the `api` and `worker` Docker targets without requiring the ML base image rebuild.

## Decision 8: RunPod Docker Image

**Decision**: The RunPod handler Dockerfile is based on `nvidia/cuda:12.1.0-runtime-ubuntu22.04` with Python 3.11, WhisperX, and pyannote installed. This is a separate image from the backend — deployed directly to RunPod's container registry.

**Rationale**: RunPod Serverless requires a self-contained Docker image with the GPU runtime. We can't reuse `zabt-worker-base` directly because RunPod has its own base image requirements, but the Python dependencies are the same.
