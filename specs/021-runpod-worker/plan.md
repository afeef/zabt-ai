# Implementation Plan: Transcription Worker Provider Switch (RunPod / Local)

**Branch**: `021-runpod-worker` | **Date**: 2026-03-08 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/021-runpod-worker/spec.md`

## Summary

Add a `TRANSCRIPTION_PROVIDER` environment variable toggle (`local` / `runpod`) that controls whether the Celery worker runs Whisper+pyannote locally or delegates transcription to a RunPod Serverless endpoint. This mirrors the existing `STORAGE_PROVIDER` (minio/s3) pattern. A new `RunPodProvider` class implements the existing `TranscriptionProvider` Protocol, and a RunPod Serverless handler (Dockerfile + handler.py) runs Whisper large-v3 + pyannote on cloud GPUs.

## Technical Context

**Language/Version**: Python 3.11 (backend worker + RunPod handler)
**Primary Dependencies**: FastAPI, Celery (Redis broker), `runpod` Python SDK (new — client calls), WhisperX, pyannote-audio, boto3
**Storage**: PostgreSQL (via SQLModel), S3/MinIO (audio files — presigned URLs passed to RunPod)
**Testing**: Manual E2E (upload meeting → verify transcription completes via RunPod). No automated E2E — this is a backend-only infrastructure feature with no user-facing UI flow.
**Target Platform**: Linux server (Contabo VPS for worker, RunPod Serverless for GPU transcription)
**Project Type**: Web service (backend worker infrastructure)
**Performance Goals**: 30-minute audio file transcribed within 5 minutes (including RunPod cold start)
**Constraints**: RunPod cold start 10-30s acceptable; polling interval 5s; job timeout 30 minutes
**Scale/Scope**: Single VPS deployment, 1-10 concurrent transcription jobs
