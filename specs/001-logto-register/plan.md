# Implementation Plan: Logto-Based User Registration

**Branch**: `001-logto-register` | **Date**: 2026-02-20 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/001-logto-register/spec.md`

## Summary

Email/password registration is broken because the backend is fully Logto-based (JWT validation
via JWKS + JIT provisioning) while the frontend still calls `POST /users/` which is (a) not
registered in the router and (b) cannot set the required `logto_id`. The fix: add a new backend
endpoint `POST /api/v1/auth/register` that creates the user in Logto via the Management API and
issues a JWT via ROPC — the custom form UI is preserved entirely. Social buttons are updated to
use a direct OIDC authorize URL with `direct_sign_in=social:{connector_id}`. No new frontend
packages are required.

## Technical Context

**Language/Version**: TypeScript 5 / Next.js 16 App Router (frontend); Python 3.11 / FastAPI (backend)
**Primary Dependencies**: `httpx` (existing — backend M2M token fetch); `logto>=0.2.1` (existing — backend Python SDK); `axios` (existing — frontend); no new npm packages required
**Storage**: PostgreSQL (User table — unchanged); Logto's internal identity datastore (external)
**Testing**: Playwright/Python (E2E — required per constitution for user-facing flows; `tests/e2e/test_registration.py`); pytest (backend unit tests if needed)
**Target Platform**: Web browser (frontend); Linux/Docker (backend)
**Performance Goals**: Registration completes in under 3 seconds (M2M token + user creation + ROPC exchange are all internal service calls)
**Constraints**: `LOGTO_M2M_APP_SECRET` MUST be a server-side env var (backend only, never exposed to frontend); ROPC grant must be enabled in Logto Console for the M2M or frontend app; custom form UI must be preserved (FR-008)
**Scale/Scope**: Single-developer setup; small user base; no concurrency concerns

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| ROPC grant (deprecated in OAuth 2.1) | Auto-sign-in after registration (FR-003) requires a JWT without a browser redirect. ROPC is the only server-side token grant that issues a user JWT given email+password. | Management API alone creates the user but cannot issue a user JWT. A redirect to /login violates FR-003. Experience API preserves the form but requires complex PKCE session management across browser calls to Logto. ROPC is deprecated but Logto supports it and it is the pragmatic choice for this server-to-server scenario. |
